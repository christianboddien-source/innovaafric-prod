1. Visión General del Ecosistema

INNOVAAFRIC es un ecosistema fintech tricontinental (África, Europa, América Latina) compuesto por 5 aplicaciones web interconectadas que comparten sesión, backend y base de datos. La arquitectura sigue el patrón JAMstack para el frontend y API REST para el backend.

Arquitectura General
Frontend estático (GitHub Pages) → API REST (Railway) → PostgreSQL (Railway) + Supabase (PostgreSQL + Storage + Auth)

3. Sistema de Sesión Compartida

Todas las aplicaciones del ecosistema comparten una única sesión almacenada en localStorage bajo la clave xm_session. Esto permite que un usuario que inicia sesión en XenderMoney pueda navegar a XenderShop, XenderDelivery o XenderBigShop sin necesidad de autenticarse nuevamente.

3.1 Estructura del Objeto de Sesión

// localStorage key: "xm_session"
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user": {
    "id": "uuid-del-usuario",
    "email": "usuario@ejemplo.com",
    "name": "Nombre Completo",
    "role": "user",            // user | staff | admin | superadmin
    "level": 1,                // 1-4
    "country": "CM",
    "kyc_status": "approved"   // pending | approved | rejected
  },
  "expires_at": "2026-07-04T00:00:00.000Z"
}
Consideración de Seguridad
El token JWT tiene una vida útil configurable. Las apps validan la expiración antes de cada operación sensible. En caso de token expirado, redirigen al login de XenderMoney.

5. Supabase — Base de Datos Frontal

Supabase sirve como capa de datos accesible directamente desde el frontend usando el SDK de JavaScript. Contiene 56+ tablas con RLS deshabilitado (acceso controlado por lógica de aplicación).

5.1 Configuración Principal

ComponenteConfiguración
RLS (Row Level Security)Deshabilitado en todas las tablas (script: supabase_rls_disable_all.sql)
Webhook Triggerpg_net HTTP trigger en tabla users — notifica al backend en cada INSERT
Storage Bucketkyc-docs — bucket público para documentos KYC
Extensión pg_netHabilitada — permite HTTP desde triggers de PostgreSQL
Tablas principales56+: users, wallets, transactions, orders, products, riders, merchants, kyc_requests, exchange_rates, notifications, api_keys, etc.

5.2 Webhook — Flujo de Registro de Usuario

1. Usuario rellena formulario en XenderMoney (frontend)
2. Frontend inserta registro en Supabase → tabla "users"
3. Trigger pg_net dispara HTTP POST a Railway API:
   POST /v1/admin/sync-user
   Headers: { "X-Webhook-Secret": "WEBHOOK_SECRET" }
   Body: { user: { id, email, name, phone, country, ... } }
4. Railway API crea/actualiza el usuario en PostgreSQL (Prisma)
5. Respuesta 200 OK confirma sincronización bidireccional

4. Shared Session System

All ecosystem applications share a single session stored in localStorage under the key xm_session. This enables single sign-on across all 5 platforms — a user authenticated in XenderMoney can seamlessly navigate to any other app without re-authentication.

FX Calculator
XenderMoney includes a currency calculator supporting 21 pairs including EUR/USD/GBP/XAF/XOF/GNF. Exchange rates are stored in the exchange_rates Supabase table and updated via the backend.
5. Security Architecture

Security is implemented at multiple layers: JWT Bearer tokens for API authentication, role-based access control via requireLevel(1-4) middleware, CORS allowlisting, HTTP security headers via helmet.js, bcrypt password hashing, and webhook secret validation for Supabase-to-Railway sync.

Access Levels
Level 1 = User (basic access) | Level 2 = Staff | Level 3 = Admin | Level 4 = Superadmin (full access)
