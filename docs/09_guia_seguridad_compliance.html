Diligencia debida y monitoreo continuo
🇬🇧 Due diligence and continuous monitoring · 🇫🇷 Diligence raisonnable et surveillance continue
🇪🇸 Español

Diligencia simplificada (SDD): usuarios con bajo riesgo, transacciones pequeñas, países de bajo riesgo.

Diligencia estándar (CDD): todos los usuarios con KYC Nivel 1. Verificación de identidad y revisión periódica anual.

Diligencia reforzada (EDD): PEPs (Personas Políticamente Expuestas), altos volúmenes, países de riesgo elevado. Revisión trimestral y fuente de fondos obligatoria.

🇬🇧 English

Simplified Due Diligence (SDD): low-risk users, small transactions, low-risk countries.

Customer Due Diligence (CDD): all users with Level 1 KYC. Identity verification and annual periodic review.

Enhanced Due Diligence (EDD): PEPs (Politically Exposed Persons), high volumes, high-risk countries. Quarterly review and mandatory source of funds verification.

🇫🇷 Français

Diligence simplifiée (SDD) : utilisateurs à faible risque, petites transactions, pays à faible risque.

Diligence standard (CDD) : tous les utilisateurs avec KYC Niveau 1. Vérification d'identité et révision périodique annuelle.

Diligence renforcée (EDD) : PPE (Personnes Politiquement Exposées), volumes élevés, pays à risque élevé. Révision trimestrielle et source de fonds obligatoire.

Sección 4
Ciberseguridad · Cybersecurity · Cybersécurité
Medidas técnicas y organizativas de seguridad de la infraestructura
🏗️
Arquitectura de seguridad técnica
🇬🇧 Technical security architecture · 🇫🇷 Architecture de sécurité technique
🔑 Autenticación
  • JWT tokens con expiración de 24h
  • Refresh tokens seguros (httpOnly cookie)
  • 2FA obligatorio para administradores
  • OTP SMS/email para transacciones >€500
  • Rate limiting: 5 intentos fallidos → bloqueo temporal 15 min
🔒 Cifrado
  • TLS 1.3 en todas las comunicaciones
  • AES-256 para datos en reposo (Supabase)
  • Contraseñas: bcrypt con salt (cost 12)
  • Claves API: almacenadas en Railway env vars
  • Documentos KYC: bucket privado Supabase Storage
🌐 Perímetro
  • CORS configurado para dominios autorizados únicamente
  • IP Whitelist para acceso al panel de administración
  • WAF (Web Application Firewall) en Railway
  • Headers de seguridad: CSP, HSTS, X-Frame-Options
  • RLS desactivado — seguridad gestionada en API layer
📊 Monitoreo
  • Logs inmutables de todas las acciones de administradores
  • Alertas en tiempo real por actividad sospechosa
  • Uptime monitoring: alertas si latencia >500ms
  • Backups automáticos Supabase: diarios, retención 30 días
  • Escaneo de vulnerabilidades mensual

Sección 5
Respuesta a Incidentes · Incident Response · Réponse aux Incidents
Procedimiento de actuación ante brechas de seguridad y fraude
🔥
Protocolo de respuesta — Brecha de datos
🇬🇧 Response protocol — Data breach · 🇫🇷 Protocole de réponse — Violation de données
1
🕐 0–1h — Detección y contención
El sistema de monitoreo detecta la anomalía y genera alerta automática al SuperAdmin. El equipo de seguridad activa el modo de contención: aislamiento del sistema afectado, revocación de tokens de acceso comprometidos y activación del plan de continuidad. // The monitoring system detects the anomaly and generates automatic alert. Security team activates containment: isolation of affected system, revocation of compromised access tokens. // Le système de surveillance détecte l'anomalie. L'équipe sécurité active le confinement : isolation du système, révocation des tokens compromis.
2
🕑 1–24h — Evaluación y notificación
Evaluación del alcance: qué datos se vieron afectados, cuántos usuarios, qué tipo de información. Si la brecha afecta a ciudadanos europeos, la AEPD debe ser notificada en máximo 72 horas. Si hay riesgo para los usuarios, estos son notificados de forma individual. // Assessment: which data affected, how many users. GDPR breach must be reported to DPA within 72 hours. // Évaluation de la portée. Si la violation concerne des citoyens européens, la CNIL doit être notifiée dans un délai de 72 heures.
3
🕓 24–72h — Investigación y remediación
Análisis forense para determinar el vector de ataque. Aplicación de parches y medidas correctivas. Revisión y endurecimiento de todos los sistemas relacionados. Documentación completa del incidente para el expediente regulatorio. // Forensic analysis to determine attack vector. Patch application and corrective measures. Full incident documentation. // Analyse forensique pour déterminer le vecteur d'attaque. Application de correctifs. Documentation complète de l'incident.
4
📋 Post-incidente — Lecciones aprendidas
Informe completo del incidente (RCA — Root Cause Analysis). Actualización de procedimientos de seguridad. Formación adicional del equipo si aplica. Notificación de cierre a las autoridades reguladoras. Comunicación final a los usuarios afectados si necesario. // Full incident report (RCA). Security procedure updates. Regulatory authority closure notification. // Rapport complet sur l'incident. Mise à jour des procédures de sécurité. Notification de clôture aux autorités réglementaires.
