0. Prerequisitos
HerramientaVersión mínimaPropósito
Node.js18+Ejecutar el backend localmente y scripts de migración
npm9+Gestor de paquetes
Git2.xControl de versiones y trigger de despliegue
Cuenta Railway—Hosting del backend API y PostgreSQL
Cuenta Supabase—Base de datos frontal, Storage, Realtime
Cuenta GitHub—Repositorios + GitHub Pages para frontend
Tiempo estimado de despliegue completo
Primera vez: ~2-3 horas. Redespliegue desde cero: ~45 minutos. El tiempo más largo lo ocupa la configuración manual de Supabase.
1
Clonar Repositorios
# Backend API
git clone https://github.com/INNOVAAFRIC/innovaafric-api.git
cd innovaafric-api
npm install

# Frontend (todas las apps)
git clone https://github.com/INNOVAAFRIC/innovaafric-prod.git
cd innovaafric-prod
# No requiere npm install — es HTML/CSS/JS puro
Estructura del repositorio backend
src/routes/ — 49 módulos de rutas | prisma/ — schema y migraciones | src/middleware/ — auth, logging | server.js — punto de entrada
2
Configurar Supabase
  1. Accede a supabase.com y crea un nuevo proyecto. Elige una región cercana a África Occidental (ej. Frankfurt o São Paulo).
  2. En el SQL Editor, ejecuta el script completo de migración:
    -- Pega y ejecuta el contenido de:
supabase_migration_v1.sql
-- Crea las 56+ tablas con índices y claves foráneas
  3. Desactiva RLS en todas las tablas:
    -- Ejecuta en SQL Editor:
supabase_rls_disable_all.sql
  4. Crea el trigger de webhook pg_net:
    supabase_webhook_trigger.sql
-- Antes de ejecutar, edita la URL del webhook:
-- Sustituye 'TU_RAILWAY_URL' por la URL real de Railway
  5. Carga los tipos de cambio iniciales:
    supabase_exchange_rates.sql
-- Inserta 21 pares EUR/USD/GBP/XAF/XOF/GNF
  6. Crea el Storage bucket para documentos KYC:
    Storage → New bucket → Nombre: kyc-docs → Marcar como Public → Crear
  7. Copia los valores de conexión:
    Settings → API → copia Project URL y service_role key
service_role key vs anon key
Usa la service_role key (no la anon key) para el backend Railway. La anon key se usa en el frontend. La service_role bypassa RLS y tiene acceso total.
3
Configurar Railway (Backend API)
  1. Accede a railway.app y crea un nuevo proyecto.
  2. Selecciona "Deploy from GitHub repo" → conecta la cuenta de GitHub → selecciona innovaafric-api.
  3. Añade el plugin PostgreSQL:
    New → Database → Add PostgreSQL → Railway inyecta automáticamente DATABASE_URL
  4. Configura las variables de entorno en Railway (Settings → Variables):
JWT_SECRETrequerida Secreto para firmar tokens JWT. Usa una cadena aleatoria de 64+ caracteres.
DATABASE_URLauto Inyectada automáticamente por el plugin PostgreSQL de Railway.
SUPABASE_URLrequerida URL del proyecto Supabase. Ej: https://xxxx.supabase.co
SUPABASE_SERVICE_KEYrequerida service_role key de Supabase (NO la anon key).
WEBHOOK_SECRETrequerida Secreto compartido entre Supabase pg_net trigger y Railway. Genera uno aleatorio.
ALLOWED_ORIGINSrequerida URLs del frontend separadas por coma. Ej: https://tu-usuario.github.io,https://xendermoney.com
NODE_ENV Establece a production
PORT Opcional. Railway lo asigna automáticamente si no se especifica.
  1. Railway ejecuta automáticamente al desplegar:
    npm install
npx prisma generate
npx prisma migrate deploy
node server.js  # o el start script de package.json
  2. Espera ~2-3 minutos. Verifica en los logs: "Server running on port XXXX"
Healthcheck de Railway
Railway verifica que el servidor responda en el puerto asignado. El servidor debe arrancar antes de que terminen las migraciones si es el primer deploy — el código está diseñado para arrancar inmediatamente mientras las migraciones corren en background.
4
Configurar Frontend (GitHub Pages)
  1. En el repositorio innovaafric-prod, abre el archivo index.html de cada app.
  2. Actualiza las constantes de Supabase al inicio del script:
    // En cada index.html de las 4 apps:
const SB_URL = 'https://TU-PROYECTO.supabase.co';
const SB_KEY = 'TU-ANON-KEY';  // anon key para el frontend
const API_URL = 'https://TU-APP.up.railway.app'; // URL de Railway
  3. Habilita GitHub Pages:
    GitHub repo → Settings → Pages → Source: Deploy from branch → Branch: main → Guardar
  4. Haz el primer push a main:
    git add .
git commit -m "feat: configure production URLs"
git push origin main
  5. GitHub Pages desplegará automáticamente en ~30 segundos.
5
Verificación del Despliegue
  1. Comprueba que Railway responde:
    curl https://TU-APP.up.railway.app/v1/auth/me
# Debe devolver: 401 Unauthorized (la API está viva)
  2. Registra un usuario de prueba desde XenderMoney.
  3. Verifica en Supabase Dashboard → Table Editor → users que el usuario fue insertado.
  4. Verifica en Railway logs que el webhook fue procesado:
    Received sync-user webhook
User created in Railway DB: uuid-del-usuario
  5. Inicia sesión con el usuario de prueba y verifica que el token JWT se almacena en localStorage.xm_session.
  6. Navega a XenderShop — debes seguir autenticado sin volver a hacer login.
Despliegue exitoso
Si todos los pasos anteriores funcionan, la infraestructura está completamente operativa.

7. Solución de Problemas
Error: prisma migrate deploy failed — relation "User" already exists
Causa: Las migraciones ya fueron aplicadas en un deploy anterior. Solución: Este error es inofensivo — el servidor arranca de todas formas. Si persiste, verifica que DATABASE_URL apunta a la base de datos correcta.
CORS error: blocked by CORS policy — No 'Access-Control-Allow-Origin'
Causa: La URL del frontend no está en ALLOWED_ORIGINS. Solución: Añade la URL exacta de GitHub Pages (incluyendo https://) a la variable ALLOWED_ORIGINS en Railway y redespliega.
401 Unauthorized — JsonWebTokenError: invalid signature
Causa: El JWT_SECRET en Railway cambió y los tokens existentes son inválidos. Solución: Los usuarios deben volver a iniciar sesión. Si ocurre en deploy nuevo, verifica que JWT_SECRET es exactamente igual al anterior.
Webhook sync-user: 403 Forbidden — Invalid webhook secret
Causa: El WEBHOOK_SECRET en el trigger pg_net de Supabase no coincide con la variable en Railway. Solución: Edita la función trigger en Supabase SQL Editor y actualiza el valor del header X-Webhook-Secret.
ERROR: extension "pg_net" does not exist
Causa: La extensión pg_net no está habilitada en el proyecto Supabase. Solución: Ve a Supabase → Database → Extensions → busca "pg_net" → habilitar. Luego vuelve a ejecutar el script del trigger.
Railway healthcheck failed — Service not responding on port
Causa: El servidor tarda demasiado en arrancar (migraciones largas). Solución: El código debe iniciar el servidor HTTP antes de ejecutar las migraciones. Verifica que el package.json start script arranque el servidor primero.

ToolMin VersionPurpose

npm9+Package manager

Railway account—Backend API hosting and PostgreSQL
Supabase account—Frontend database, Storage, Realtime

1
Clone Repositories
# Backend API
git clone https://github.com/INNOVAAFRIC/innovaafric-api.git
cd innovaafric-api && npm install

# Frontend (all apps — pure HTML/CSS/JS, no npm needed)
git clone https://github.com/INNOVAAFRIC/innovaafric-prod.git
2
Supabase Setup
  1. Create a new project at supabase.com. Choose a region close to West Africa (Frankfurt or São Paulo recommended).
  2. In the SQL Editor, run supabase_migration_v1.sql — creates all 56+ tables.
  3. Run supabase_rls_disable_all.sql — disables Row Level Security on all tables.
  4. Edit and run supabase_webhook_trigger.sql — replace 'TU_RAILWAY_URL' with your actual Railway URL before running.
  5. Run supabase_exchange_rates.sql — seeds initial FX rates for 21 currency pairs.
  6. Create Storage bucket: Storage → New bucket → Name: kyc-docs → Check Public → Create.
  7. Copy credentials: Settings → API → copy Project URL and service_role key.
3
Railway Setup (Backend API)
  1. Create a new project at railway.app → Deploy from GitHub repo → select innovaafric-api.
  2. Add PostgreSQL plugin: New → Database → Add PostgreSQL → DATABASE_URL is auto-injected.
  3. Set all required environment variables (see reference table in Spanish section).
  4. Railway automatically runs: npm install → prisma generate → prisma migrate deploy → node server.js
  5. Wait ~2-3 minutes. Check logs for: "Server running on port XXXX"
4
Frontend Configuration (GitHub Pages)
  1. In each app's index.html, update the configuration constants at the top of the script block with your Supabase URL, anon key, and Railway API URL.
  2. Enable GitHub Pages: repo Settings → Pages → Source: Deploy from branch → Branch: main → Save.
  3. Push to main — GitHub Pages deploys automatically in ~30 seconds.
5
Deployment Verification
  1. Test API is alive: GET /v1/auth/me → must return 401 (not 502 or timeout).
  2. Register a test user on XenderMoney.
  3. Check Supabase Table Editor → users table — user should appear.
  4. Check Railway logs — should show webhook received and user synced to Railway DB.
  5. Log in with test user → verify localStorage.xm_session is populated.
  6. Navigate to XenderShop — should still be authenticated (shared session working).
Deployment successful
All 6 checks passing means the full infrastructure is operational.
